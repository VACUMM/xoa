Using the xoa.cf module¶

Introduction¶

This module is an application and extension of a subset of the CF conventions. It has two intents:

  • Searching xarray.DataArray variables or coordinates with another xarray.DataArray or a xarray.Dataset, by scanning name and attributes like units and standard_name.

  • Formatting xarray.DataArray variables or coordinates with unique name and standard_name and long_name attributes, with support of staggered grid location syntax.

Accessing the current specifications¶

Scanning and formatting actions are based on specifications, and this module natively includes a default configuration for various oceanographic, sea surface and atmospheric variables and coordinates. A distinction is made between data variables data_vars and coordinates coords, like in xarray.

Getting the current specifications for data variables and coordinates with the get_cf_specs() function:

In [1]: from xoa import cf

In [2]: cfspecs = cf.get_cf_specs()

In [3]: cfspecs["data_vars"] is cfspecs.data_vars
Out[3]: True

In [4]: cfspecs["data_vars"] is cfspecs.coords
Out[4]: False

In [5]: cfspecs.data_vars.names[:3]
Out[5]: ['temp', 'ptemp', 'sal']

In [6]: cfspecs.coords.names[:3]
Out[6]: ['time', 'lon', 'lat']

See the appendix The default configuration for the list of available default specifications.

Searching within a Dataset or DataArray¶

Let’s define a minimal dataset:

In [11]: nx = 3

In [12]: lon = xr.DataArray(np.arange(3, dtype='d'), dims='mylon',
   ....:     attrs={'standard_name': 'longitude'})
   ....: 

In [13]: temp = xr.DataArray(np.arange(20, 23, dtype='d'), dims='mylon',
   ....:     coords={'mylon': lon},
   ....:     attrs={'standard_name': 'sea_water_temperature'})
   ....: 

In [14]: sal = xr.DataArray(np.arange(33, 36, dtype='d'), dims='mylon',
   ....:     coords={'mylon': lon},
   ....:     attrs={'standard_name': 'sea_water_salinity'})
   ....: 

In [15]: ds = xr.Dataset({'mytemp': temp, 'mysal': sal})

All these arrays are CF compliant according to their standard_name attribute, despite their name is not really explicit.

Check if they match known or explicit CF items:

In [16]: cfspecs.coords.match(lon, "lon") # explicit
Out[16]: True

In [17]: cfspecs.coords.match(lon, "lat") # explicit
Out[17]: False

In [18]: cfspecs.coords.match(lon) # any known
Out[18]: 'lon'

In [19]: cfspecs.data_vars.match(temp) # any known
Out[19]: 'temp'

In [20]: cfspecs.data_vars.match(sal) # any known

Search for known CF items:

In [21]: mytemp = cfspecs.search(ds, "temp")

In [22]: mylon = cfspecs.search(mytemp, "lon")

Datasets are searched for data variables (“data_vars”) and data variables are searched for coordinates (“coords”). You can also search for coordinates in datasets, for instance like this:

In [23]: cfspecs.coords.search(ds, "lon")
Out[23]: 
<xarray.DataArray 'mylon' (mylon: 3)>
array([0., 1., 2.])
Coordinates:
  * mylon    (mylon) float64 0.0 1.0 2.0
Attributes:
    standard_name:  longitude

See also

  • CF items: lon lat temp sal

  • Methods: xoa.cf.CFCoordSpecs.match() xoa.cf.CFVarSpecs.match() xoa.cf.CFSpecs.search() xoa.cf.CFCoordSpecs.search() xoa.cf.CFVarSpecs.search()

Formatting¶

It is possible to format, or even auto-format data variables and coordinates.

During an auto-formatting, each array is matched against CF specs, and the array is formatting when a matching is successfull. If the array contains coordinates, the same process is applied on them, as soon as the format_coords keyword is True.

Explicit formatting:

In [24]: cfspecs.format_coord(lon, "lon")
Out[24]: 
<xarray.DataArray 'lon' (mylon: 3)>
array([0., 1., 2.])
Dimensions without coordinates: mylon
Attributes:
    standard_name:  longitude
    units:          degrees_east
    long_name:      Longitude

In [25]: cfspecs.format_data_var(temp, "temp")
Out[25]:

Using the accessors¶

Accessors for xarray.Dataset and xarray.DataArray can be registerd with the xoa.cf.register_cf_accessors():

In [29]: cf.register_cf_accessors()

These accessors make it easy to use some of the xoa.cf.CFSpecs capabilities. Here are some example of usage:

In [30]: temp
Out[30]: 
<xarray.DataArray (mylon: 3)>
array([20., 21., 22.])
Coordinates:
  * mylon    (mylon) float64 0.0 1.0 2.0
Attributes:
    standard_name:  sea_water_temperature

In [31]: temp.cf.get("lon")
Out[31]: 
<xarray.DataArray 'mylon' (mylon: 3)>
array([0., 1., 2.])
Coordinates:
  * mylon    (mylon) float64 0.0 1.0 2.0
Attributes:
    standard_name:  longitude

In [32]: ds.cf.get("temp")
Out[32]: 
<xarray.DataArray 'mytemp' (mylon: 3)>
array([20., 21., 22.])
Coordinates:
  * mylon    (mylon) float64 0.0 1.0 2.0
Attributes:
    standard_name:  sea_water_temperature

In [33]: ds.cf.lon
Out[33]:

As you can see, accessing an accessor attribute or item make an implicit call to get. The root accessor attr:cf agive accessor to two sub-accessors, data_vars and coords, for being able to specialize the searches.

Changing the CF specs¶

Default user file¶

The xoa.cf module has internal defaults as shown in appendix The default configuration.

You can extend these defaults with a user file, whose location is printable with the following command, at the line containing “user CF specs file”:

$ xoa info paths
- xoa library dir: /local/home/sraynaud/Src/xoa/xoa
- user config file: /local/home/sraynaud/.local/share/xoa/xoa.cfg [*]
- user CF specs file: /local/home/sraynaud/.config/xoa/cf.cfg [*]
- user CF cache file: /local/home/sraynaud/.cache/xoa/cf.pyk
- data samples: croco.south-africa.zonal.nc croco.south-africa.surf.nc
*: file not present

Update the current specs¶

The current specs can be updated with different methods.

From a well structured dictionary:

In [41]: cfspecs.load_cfg({"data_vars": {"banana": {"standard_name": "banana"}}})

In [42]: cfspecs.data_vars["banana"]
Out[42]: 
{'standard_name': ['banana'],
 'name': ['banana'],
 'long_name': ['Banana'],
 'units': [],
 'domain': 'generic',
 'cmap': None,
 'inherit': None,
 'squeeze': None,
 'search_order': 'sn',
 'select': {},
 'processed': True}

From a configuration file: instead of the dictionary as an argument to load_cfg() method, you can give either a file name or a multi-line string with the same content as the file. Following the previous example:

[data_vars]
    [[banana]]
        standard_name: banana

If you only want to update a category, you can use such method (here set_specs()):

In [43]: cfspecs.data_vars.set_specs("banana", name="bonono")

In [44]: cfspecs.data_vars["banana"]["name"]
Out[44]: ['banana', 'bonono']

Alternatively, a xoa.cf.CFSpecs instance can be loaded with the load_cfg() method, as explained below.

Create new specs from scratch¶

To create new specs, you must instantiate the xoa.cf.CFSpecs class, with an input type as those presented above:

  • A config file name.

  • A Multi-line string in the format of a config file.

  • A dictionary.

  • A configobj.ConfigObj instance.

  • Another CFSpecs instance.

  • A list of them, with the having priority over the lasts.

The initialization also accepts two options:

  • default: wether to load or not the default internal config.

  • user: wether to load or not the user config file.

An config created from default and user configs:

In [45]: banana_specs = {"data_vars": {"banana": {"standard_name": "banana"}}}

In [46]: mycfspecs = cf.CFSpecs(banana_specs)

In [47]: mycfspecs["data_vars"]["sst"]["standard_name"]
Out[47]: ['sea_surface_temperature', 'surface_sea_water_temperature']

In [48]: mycfspecs["data_vars"]["banana"]["standard_name"]
Out[48]: ['banana']

An config created from scratch:

In [49]: mycfspecs = cf.CFSpecs(banana_specs, default=False, user=False)

In [50]: mycfspecs.pprint(depth=2)
{'accessors': {'properties': {...}},
 'coords': {},
 'data_vars': {'banana': {...}},
 'dims': {'f': [], 't': [], 'x': [], 'y': [], 'z': []},
 'sglocator': {'name_format': '{root}_{loc}', 'valid_locations': None}}

An config created from two other configs:

In [51]: cfspecs_banana = cf.CFSpecs(banana_specs, default=False, user=False)

In [52]: apple_specs = {"data_vars": {"apple": {"long_name": "Big apple"}}}

In [53]: cfspecs_apple = cf.CFSpecs(apple_specs, default=False, user=False)

In [54]: cfspecs_fruits = cf.CFSpecs([cfspecs_apple, cfspecs_banana],
   ....:     default=False, user=False)
   ....: 

In [55]: cfspecs_fruits.data_vars.names
Out[55]: ['apple', 'banana']

Replacing the currents CF specs¶

As shown before, the currents CF specs are accessible with the xoa.cf.get_cf_specs() function. You can replace them with the xoa.cf.set_cf_specs class, to be used as a fonction.

In [56]: cfspecs_old = cf.get_cf_specs()

In [57]: cf.set_cf_specs(cfspecs_banana)
Out[57]: <xoa.cf.set_cf_specs at 0x7ff0f9c6f350>

In [58]: cf.get_cf_specs() is cfspecs_banana
Out[58]: True

In [59]: cf.set_cf_specs(cfspecs_old)
Out[59]: <xoa.cf.set_cf_specs at 0x7ff0f9c6fdd0>

In [60]: cf.get_cf_specs() is cfspecs_old
Out[60]: True

In case of a temporary change, you can used set_cf_specs in a context statement:

In [61]: with cf.set_cf_specs(cfspecs_banana) as myspecs:
   ....:     print('inside', cf.get_cf_specs() is cfspecs_banana)
   ....:     print('inside', myspecs is cf.get_cf_specs())
   ....: 
inside True
inside True

In [62]: print('outside', cf.get_cf_specs() is cfspecs_old)
outside True

Application with an accessor usage:

In [63]: data = xr.DataArray([5], attrs={'standard_name': 'sea_surface_banana'})

In [64]: ds = xr.Dataset({'toto': data})

In [65]: mycfspecs = cf.CFSpecs({"data_vars": {"ssb":
   ....:     {"standard_name": "sea_surface_banana"}}})
   ....: 

In [66]: with cf.set_cf_specs(mycfspecs):
   ....:     print(ds.cf.get("ssb"))
   ....: 
<xarray.DataArray 'toto' (dim_0: 1)>
array([5])
Dimensions without coordinates: dim_0
Attributes:
    standard_name:  sea_surface_banana
